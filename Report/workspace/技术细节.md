# B-BOT Report 技术细节整理

用途：作为修改 `Report/main.tex` 的技术事实库。这里不追求优美叙述，而是把硬件、PCB、嵌入式控制、蓝牙、WiFi TCP、ROS 2 摄像头模块和实验设计的关键事实整理清楚，避免 report 中出现事实错误或遗漏原创性证据。

更新时间：2026-04-26

---

## 0. 核心事实边界

### 必须写清楚

- B-BOT 是一个自研轮腿自平衡机器人平台。
- 机器人机械平台不是 Yahboom 底盘，也不是 Yahboom 轮腿机器人改装。
- ESP32 控制板是本项目自行绘制的 custom PCB。
- 自研 ESP32 PCB 包含：
  - ESP32 主控。
  - IMU 路径。
  - 六个电机接口。
  - 一个额外拓展接口。
  - 电源管理。
- Yahboom 只作为 ROS 2 WiFi 摄像头模块来源，不应被写成机器人平台来源。
- 底盘 ESP32 控制器当前不是 ROS 2/micro-ROS 节点。
- micro-ROS 只用于摄像头模块把图像送入主机 ROS 2 图。
- 真实机器人控制命令走 PC/host 到 ESP32 的 WiFi TCP line protocol。

### 不应该再出现的错误表述

- 不要写：the robot platform is based on a Yahboom ESP32 wheel-legged robot。
- 不要写：Yahboom base code / Yahboom base platform 是机器人控制或机械平台基础。
- 不要把 PlatformIO 的 `esp32doit-devkit-v1` 环境写成最终硬件就是商业 ESP32 dev board。它只是兼容 ESP32 Arduino 的 build target。
- 不要把视觉链路写成底层闭环控制。视觉只能是 supervisory teleoperation。

### 目前 report 主线

> Self-designed wheel-legged robot platform + custom ESP32 controller PCB + local ESP32 real-time LQR/PID/VMC balance control + safety-aware multi-source teleoperation + ROS 2 WiFi camera vision interface.

## 1. 系统总体架构

### 数据流

```text
Self-designed wheel-legged robot
  -> custom ESP32 PCB
  -> FreeRTOS firmware
  -> CAN motor feedback + MPU6050 attitude + leg kinematics
  -> 4 ms local LQR/PID/VMC control loop
  -> CAN motor commands

Xbox controller
  -> BLE HID notification
  -> ESP32 target update
  -> local control loop

PC keyboard / host scripts
  -> WiFi TCP :23
  -> ESP32 line command parser
  -> direct target or queued action
  -> local control loop

Yahboom ROS 2 WiFi camera module
  -> WiFi + micro-ROS agent
  -> /espRos/esp32camera
  -> host ROS 2 vision bridge + MediaPipe
  -> WiFi TCP :23
  -> ESP32 line command parser
  -> local control loop
```

### 核心设计判断

- 平衡闭环必须留在 ESP32 本地，因为 4 ms 控制环不能依赖 WiFi、摄像头帧率、ROS 2 调度或 MediaPipe 推理。
- BLE、UART、WiFi TCP 和视觉命令都只更新 `target` 或加入队列，不直接替代姿态反馈控制。
- 这使 report 的重点不是“接了 ROS 摄像头”，而是“非实时远程输入如何安全接入实时不稳定机器人”。

这个顺序可以在 Methods 中解释为：先启动通信和底层传感/电机，再启动腿部状态估计和控制任务，最后启动遥控输入。

## 4. 控制算法细节

### 控制目标

文件：`include/ctrl.h`

`Target` 结构包含：

- `position`
- `speedCmd`
- `speed`
- `yawSpeedCmd`
- `yawAngle`
- `rollAngle`
- `legLength`
- `target`
- `crossStepEnabled`

`StateVar` 结构包含：

- `theta`
- `dTheta`
- `x`
- `dx`
- `phi`
- `dPhi`

主控制器并不是直接使用遥控输入，而是使用经过平滑和限制后的 `target`。

### 控制状态向量

文件：`src/ctrl.cpp`

主 LQR 状态：

```text
x = [
  theta,    // virtual leg/body lean state
  dTheta,
  wheel position x,
  wheel speed dx,
  body pitch phi,
  pitch rate dPhi
]
```

其中：

- `phi = imuData.pitch`
- `dPhi = imuData.pitchSpd`
- `x = average wheel angle * wheelRadius`
- `dx = average wheel speed * wheelRadius`
- `theta = average leg angle - pi/2 - pitch`
- `dTheta = average leg angular speed - pitch speed`
- `wheelRadius = 0.0325 m`

### LQR gain scheduling

文件：

- `src/matlab_code/lqr_k.c`
- `include/matlab_code/lqr_k.h`

特点：

- `lqr_k(legLength, kRes)` 根据当前平均腿长 `legLength` 生成 12 个 LQR gain。
- 这些 gain 是 MATLAB Coder 生成，文件头显示生成日期为 2026-02-10。
- 这是 report 中“轮腿系统不是固定倒立摆”的关键技术点：腿长改变质心和等效动力学，因此 LQR gain 与腿长相关。

可用于 report 的论证：

> The controller uses leg-length-dependent LQR gains generated from the modelling workflow, rather than a single fixed gain matrix. This directly addresses the changing operating point introduced by the wheel-legged morphology.

### Target ramp 和限幅

文件：`src/ctrl.cpp`, `Ctrl_TargetUpdateTask`

关键逻辑：

- 控制目标每 4 ms 更新。
- 速度命令 `speedCmd` 不直接进入 LQR，而是通过斜坡变成 `target.speed`。
- 斜坡步长与腿长有关：

```c
speedSlopeStep = -(legLength - 0.07f) * 0.02f + 0.002f;
```

- `target.position += target.speed * 0.004f`
- 位置目标限制在当前实际位置的 `±0.1 m`。
- 速度目标限制在当前实际速度的 `±0.3 m/s`。
- `yawAngle += yawSpeedCmd * 0.004f`

Report 解释：

- 这不是简单遥控映射，而是 command shaping。
- 它降低阶跃命令导致的 pitch spike。
- E4/E9 可以用来验证 target ramp 的价值。

### PID 回路

文件：`src/ctrl.cpp`, `src/pid.cpp`

| 回路 | 用途 | 当前参数 |
|---|---|---|
| Roll PID | 横滚姿态调节，通过左右腿力差补偿 | inner `P=0.5, D=2.5, max=5`; outer `P=10, max=4` |
| Yaw PID | 偏航目标跟踪和转向 | inner `P=0.015, max=0.3`; outer `P=15, max=6` |
| Leg length PID | 平均腿长调节 | inner `P=10, I=1, D=30, maxI=2, max=10`; outer `P=5, max=0.5` |
| Leg angle diff PID | 左右腿角差控制/交叉步辅助 | inner `P=0.04, max=1`; outer `P=12, max=20` |

`PID_CascadeCalc()` 是串级 PID：外环根据位置/角度误差产生内环目标，内环根据速度反馈输出。

### VMC: virtual model control

文件：

- `src/ctrl.cpp`
- `src/matlab_code/leg_vmc_conv.c`

控制器先计算虚拟腿力和虚拟腿力矩：

- `leftForce`
- `rightForce`
- `leftTp`
- `rightTp`

然后调用：

```c
leg_vmc_conv(force, torque, joint_angle_1, joint_angle_0, jointTorque)
```

把虚拟腿部力/力矩映射为两个关节电机扭矩命令。

Report 解释：

- LQR 负责平衡方向的耦合状态反馈。
- PID 负责 roll/yaw/leg length 等辅助目标。
- VMC 把控制器的虚拟腿部命令转换为实际关节扭矩。

### Ground detection 和 airborne safety

文件：`src/ctrl.cpp`

关键参数：

- 腿部质量 `legMass = 0.08 kg`
- 通过左右腿力、重力项、腿长加速度和垂直加速度估计支撑力。
- 双腿支撑力都大于 `3 N` 时判断触地。
- 若短时间离地但距离上次触地小于 `1 s`，仍认为触地，避免弹跳误判。
- 从离地转为触地时进入 cushioning 状态，并重置 `target.position = stateVar.x`。
- 离地时关闭轮电机，并修改 LQR gain，使控制器更偏向保持腿姿态。

Report 价值：

- 这是轮腿平台区别于普通两轮平衡车的重要部分。
- 可以作为安全/鲁棒性设计点，而不是只讲 LQR。

### Balance enable / zero torque safety

文件：`src/ctrl.cpp`

默认：

```c
bool balanceEnabled = false;
```

当 `balanceEnabled == false`：

- 若不是 `legTestMode`，六个电机全部输出 0。
- 清空 `speedCmd` / `yawSpeedCmd`。
- 控制计算跳过。

Report 解释：

- 上电默认不是自动站立。
- 平衡必须由 Y/LT/serial command 等明确启用。
- 这是调试和实验安全边界。

### CAN 通信

文件：`src/can.cpp`

| 项 | 值 |
|---|---|
| ESP32 CAN/TWAI TX | GPIO33 |
| ESP32 CAN/TWAI RX | GPIO32 |

电机反馈 CAN IDs：

| ID | 电机 |
|---|---|
| `0x101` | leftJoint[0] |
| `0x102` | leftJoint[1] |
| `0x103` | leftWheel |

| `0x107` | rightWheel |

### 电机输出

文件：`src/motor.cpp`

系统有六个电机对象：

- `leftJoint[0]`
- `leftJoint[1]`
- `leftWheel`
- `rightJoint[0]`
- `rightJoint[1]`
- `rightWheel`

每 2 ms：

1. `Motor_UpdateVoltage()` 根据目标扭矩、转速和反电动势补偿计算输出电压。
2. 电压被限幅。
3. 左右两组通过 CAN 发送。

初始化参数：

| 电机类型 | maxVoltage | torqueRatio | 备注 |
|---|---:|---:|---|
| 关节电机 | 9.69 V | 0.0333 | 使用 `Motor_CalcRevVolt4310()`，代码注释中实际曲线改为 4010 版本 |
| 轮电机 | 4.0 V | 0.01 | 使用 `Motor_CalcRevVolt2805()` |

### 电机校准

文件：`src/motor.cpp`

功能：

- 使用 `Preferences` 保存关节电机 offset 和 dir。
- 若没有保存参数，初始化流程会：
  - 逐个关节电机施加小扭矩，人工确认方向。
  - 等待用户把腿放到水平位置并发送 `SET`。
  - 计算零点 offset。
  - 保存参数。

Report 用途：

- 可以在 Methods 中简短说明 calibration is stored in non-volatile preferences。
- 不需要正文展开全部交互流程，除非 Appendix H/软件证据需要。

## 6. IMU、腿部运动学和电源监测

### IMU

文件：`src/imu.cpp`

| 项 | 值 |
|---|---|
| IMU | MPU6050 |
| I2C pins | GPIO21 SDA, GPIO22 SCL |
| I2C clock | 400 kHz |
| 姿态算法 | MPU6050 DMP |
| 任务周期 | 5 ms |
| 加速度量程 | `MPU6050_ACCEL_FS_16` |

处理：

- 启动时从 ESP32 `Preferences` 加载已保存 offset。
- 若没有 offset，则自动校准并保存。
- 输出：
  - `roll`, `pitch`, `yaw`
  - `rollSpd`, `pitchSpd`, `yawSpd`
  - `zAccel`
- yaw 使用 wrap-around counter 展开，避免 `±pi` 跳变。

### 腿部运动学

文件：

- `src/legs.cpp`
- `src/matlab_code/leg_position.c`
- `src/matlab_code/leg_speed.c`

每 4 ms：

- 用两个关节角计算虚拟腿长 `length` 和虚拟腿角 `angle`。
- 用关节速度和角度计算 `dLength` 和 `dAngle`。
- 通过差分和低通滤波估计 `ddLength`。

Report 解释：

- 自平衡控制不是直接基于单个关节角，而是基于虚拟腿状态。
- 这使得 LQR/VMC 可以在更高层的轮腿变量上工作。

### 电源监测

文件：`src/adc.cpp`

功能：

- ADS1115 读取 channel 0。
- 电压估算：`ADS1115.computeVolts(adc0)/0.191678`。
- 若电压在 14--17 V 之间，更新：

```c
motorOutRatio = (16.8f - vot) / VOLTAGE_COMPENSATION + 1.0f;
```

其中 `VOLTAGE_COMPENSATION = 10.0f`。

Report 可写法：

- Battery voltage is monitored for diagnostics and future compensation.
- 当前结果不应夸大为完整电池管理系统，除非 PCB 和实验都有进一步证据。

## 7. 命令系统和安全仲裁

### 两类命令路径

| 类型 | 示例 | 特点 |
|---|---|---|
| Direct teleop | `DRIVE,<speed_mm_s>,<yaw_mrad_s>`, `YAWRATE,<mrad_s>` | 绕过队列，直接更新连续目标；必须周期性刷新 |
| Queue command | `FORWARD`, `BACKWARD`, `LEFT`, `RIGHT`, `JUMP`, `CROSSLEG` | 加入动作队列，按持续时间执行 |

### Direct command 限幅

文件：`src/serial.cpp`

| 参数 | 值 |
|---|---:|
| `MAX_SPEED` | 0.8 m/s |
| `MAX_YAWSPEED` | 4.5 rad/s |

### 仲裁逻辑

关键规则：

- 当 command queue 运行或暂停时，BLE 不覆盖 target。
- Direct `DRIVE` / `YAWRATE` 在队列忙时被忽略。
- PC keyboard tool 启动时发送 `BLE_DISABLE`，退出时发送 `DRIVE,0,0`、`YAWRATE,0`、`BLE_ENABLE`。
- WiFi TCP 断线/超时时，ESP32 注入 full stop：`DRIVE,0,0` + `YAWRATE,0` + `QUEUE_STOP`。

Report 用途：

- 这是 O3 的核心：multi-source command arbitration。
- 不应只写“支持多种输入”，要写“避免多种输入同时覆盖目标量”。

## 8. WiFi TCP 命令服务器

文件：`src/wifi_cmd.cpp`

### 连接与寻址

- ESP32 使用 station 模式连接 WiFi。
- 连接超时：15 s。
- 重连间隔：5 s。
- mDNS host 默认 `wheeleg`，即 `wheeleg.local`。
- TCP server 使用 `WIFI_TCP_PORT`，当前 report 和工具按 port 23 使用。
- 使用 `TCP_NODELAY` 降低交互延迟。

### 协议

客户端连接后 ESP32 发送：

```text
HELLO,<esp_ms>,wheeleg_tcp,<ip>:<port>
```

每条命令后返回：

```text
ACK,<esp_ms>,<rc>,<line>
NACK,<esp_ms>,<rc>,<line>
```

这使 E5/E11 可以用 PC send time 到 ACK receive time 测量 command-entry acknowledgement latency。

### 安全 watchdog

| 机制 | 代码行为 | 实验对应 |
|---|---|---|
| TCP idle watchdog | 超过 `WIFI_WATCHDOG_MS` 无 line，force close + full stop | E6 idle timeout |
| client drop | TCP client 掉线后 full stop | E6 TCP close |
| WiFi lost | WiFi link lost 时 full stop | 可作为 future/final test |
| direct DRIVE watchdog | 500 ms 未刷新 `DRIVE` 则清零 | E6 direct watchdog |

当前 measured 结果：

- E5 TCP ACK latency: n=300, median 37.41 ms, p95 88.31 ms, p99 143.47 ms, max 368.89 ms, non-ACK 0。
- E6 direct DRIVE watchdog: n=10, event 10/10, median ACK-to-watchdog 517.93 ms, p95 538.81 ms。
- E6 TCP close: n=10, client-drop full-stop 10/10, median close-to-event 33.35 ms。
- E6 TCP idle: n=10, full-stop 10/10, median wait-to-EVT 1481.08 ms。

## 9. Xbox BLE 控制

文件：

- `include/XboxSeriesXControllerESP32_asukiaaa.hpp`
- `src/XboxControllerNotificationParser.cpp`

### BLE 连接方式

- 使用 NimBLE-Arduino。
- 自动扫描目标：
  - 设备名 `Xbox Wireless Controller`。
  - 或 appearance 964 + HID service。
  - 或 manufacturer data + HID service。

- 断开连接时，若 BLE input enabled 且队列不忙，则清零 `speedCmd` 和 `yawSpeedCmd`。

### report 解释重点

### 80+ 实验证据重点

要冲高分，Results 章不应只是“功能演示”。应围绕以下判断：

- ESP32 4 ms local loop 的实时性是否足够。
- LQR/PID/VMC 是否能稳定轮腿平台。
- 腿长改变是否影响稳定性，因此 gain scheduling 是否合理。
- target ramp / watchdog / dry-run / stunt gate 是否有实验依据。
- WiFi/ROS/MediaPipe 为什么只能做 supervisory teleoperation。

### Baseline / comparison 要求

Baseline 不能只靠文字暗示，需要在 Results 中明确写出“和什么比”。当前 report 已加强为四类 baseline：

| Baseline 类型 | 对应实验 | 用途 |
|---|---|---|
| 4 ms control-period target | E8 | 证明 ESP32 local loop 有实时性依据 |
| FULL vs `FIXED_LQR` / `NO_RAMP` | E9 | 证明 gain scheduling 和 target ramp 不是无依据复杂化 |
| Direct WiFi TCP path vs vision bridge path | E5 / E11 | 证明视觉层只是 supervisory，不应进入 balance loop |
| Dry-run / NoHand / false-command cases | E6 / E10 | 证明 watchdog、dry-run、stunt gate 的必要性 |

最关键的是 E9：如果最终硬件时间有限，优先拿到 FULL vs `FIXED_LQR` 或 FULL vs `NO_RAMP` 的小扰动/低速实测对照。它是支撑 O1 控制设计达到高分的最直接 baseline。

### 实验写作注意

- E5/E6/E8/E10/E11 是实测，可以作为正式证据。
- E1/E2/E3/E4b/E9 目前必须明确 planning，不能在 conclusion 中当作已实测。
- 实测失败/误触发不要隐藏，尤其 E10 的 false forward / no stable / false direction 应作为 safety layer 必要性的证据。
- E11 不是完整 human gesture to robot latency，只是 bridge command sent to ESP32 ACK。完整视觉路径还至少包含 camera frame period 和 debounce frames。

## 14. 可直接用于英文 report 的技术句子草稿

### Hardware originality

```text
The robot platform and controller electronics were designed and built as part of this project. The controller uses a custom ESP32 PCB that integrates the IMU path, six motor interfaces, an expansion interface and power-management circuitry. The Yahboom component used in the system is limited to the separate ROS 2 WiFi camera module used for the vision input path.
```

### Real-time boundary

```text
The stabilising balance loop is kept local to the ESP32. Wireless, BLE and vision inputs update command targets or queue actions, but they do not replace the 4 ms embedded feedback loop.
```

### Supervisory vision

```text
The ROS 2 and MediaPipe pipeline is treated as a supervisory teleoperation layer. Its frame rate and acknowledgement latency are useful for human commands, but they are not suitable for closing the stabilising control loop.
```

### Safety arbitration

```text
The command interface separates direct teleoperation from queued actions. Direct DRIVE and YAWRATE commands are protected by a 500 ms refresh watchdog, while TCP disconnects and idle timeouts inject a full-stop sequence consisting of DRIVE,0,0, YAWRATE,0 and QUEUE_STOP.
```

### Gain scheduling

```text
Because leg length changes the operating point of the wheel-legged system, the LQR feedback gains are generated as a function of the current virtual leg length rather than held fixed.
```
