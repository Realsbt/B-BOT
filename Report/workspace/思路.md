# B-BOT Final Report 写作思路

这篇 report 不应该按开发流水账写，而应该按“问题 → 设计 → 证据 → 反思”写。主线是：

> B-BOT is a WiFi-enabled self-balancing wheel-legged robot. The core contribution is an ESP32-based real-time balancing controller using LQR/PID/VMC, extended with safe multi-source command arbitration and a ROS 2 + MediaPipe vision teleoperation pipeline.

重点不是“用了 micro-ROS”，而是：

- 底盘控制是核心：ESP32 本地完成高频平衡控制，不依赖无线网络闭环。
- 通信和视觉是扩展层：camera micro-ROS 只负责图像进入 ROS 2 主机，真实机器人命令由 PC 通过 WiFi TCP 发送。
- 安全仲裁是工程亮点：Xbox BLE、UART 队列、WiFi TCP、视觉命令都可能影响同一套目标量，因此必须解释 BLE disable、队列优先级、direct command watchdog、TCP idle watchdog、dry-run 和 stunt armed。
- 实验结果必须服务 objectives：每个实验都回答 O1/O2/O3 中的一个问题。

最重要的一句话：

> 这篇 report 要把项目包装成一个“受限硬件上的实时控制系统 + 安全遥操作扩展”，而不是一个“ROS/micro-ROS 项目”。

### 2.2 Wheel-legged and self-balancing robotics

逻辑：

普通两轮平衡车：

- 模型相对简单。
- 主要控制 pitch 和 wheel motion。

轮腿机器人：

- 多了腿长、关节角、重心高度和触地状态。
- 平衡控制更复杂。
- 具备跳跃、跨步、变高等潜力。

代表系统：

- Grasser JOE
- two-wheeled robot review
- Ascento
- LQR + ADRC wheel-legged robot

结尾引出你的设计挑战：

- 在小型 ESP32 平台上实现实时控制。
- 同时保留可演示动作和遥操作扩展。

### 2.3 Control strategies comparison

不要只解释 PID/LQR/VMC 是什么，要比较它们在本项目里的角色。

PID：

- 简单、低成本。
- 适合腿长、roll、yaw 等局部闭环。
- 不足是难以处理整车强耦合。

LQR：

- 适合倒立摆/轮腿平衡的大闭环。
- 能把 pitch、position、velocity 等状态统一考虑。
- 依赖模型和调参。

VMC：

- 适合轮腿机构。
- 将虚拟腿力/髋部力矩映射到两个关节电机。

ADRC / 其他方法：

- 用作文献对比。
- 说明为什么本项目没有采用，例如实现复杂、调参成本高或时间限制。

结尾：

- 本项目采用 LQR 负责整车平衡。
- PID 负责辅助姿态、腿长和 yaw。
- VMC 负责力到关节力矩映射。

### 2.4 Embedded ROS, WiFi teleoperation and lossy networks

这里必须写清边界。

当前系统：

- camera micro-ROS → `/espRos/esp32camera`
- ROS 2 host → MediaPipe
- MediaPipe/bridge → WiFi TCP command
- ESP32 command parser → target variables / queue

不要写成：

- ESP32 底盘是 micro-ROS node。
- 控制命令依赖 ROS 2 QoS。
- WiFi 进入底层平衡闭环。

应该强调：

- 无线链路有延迟和断线风险。
- 所以 WiFi/视觉只作为遥操作输入。
- 底层平衡控制必须留在 ESP32。
- TCP watchdog 和断线 stop 是安全设计的一部分。

### 2.5 Summary

用一个 `Gap → Design choice` 表收束。

示例：

| Gap | Design choice |
|---|---|
| Need real-time balance on limited MCU | ESP32 FreeRTOS local control loop |
| Need wheel-legged force coordination | LQR/PID/VMC hierarchy |
| Need safe remote input | TCP watchdog + BLE/queue arbitration |
| Need vision teleop | ROS 2 + MediaPipe on PC, not on ESP32 |

### 3.4 Perception and state estimation

写实际实现。

IMU：

- MPU6050 DMP 输出 yaw/pitch/roll。
- Gyro 转换成 rad/s。
- Yaw unwrap 处理跨 ±π。
- IMU offset calibration 保存到 flash。

Leg kinematics：

- `leg_position` 计算腿长和腿角。
- `leg_speed` 计算腿长速度和腿角速度。
- 腿长加速度用低通滤波。
- 这些来自 MATLAB generated code。

- 不要在 Methods 里写你用了 Madgwick/Mahony，除非只是文献背景。
- Methods 必须写实际代码实现。

### 3.5 Control architecture and algorithms

推荐顺序。

1. State definition

LQR 状态：

```text
theta, dtheta, x, dx, phi, dphi
```

说明每个状态来自哪里。

2. Target update

- `target.speedCmd` 经斜坡变成 `target.speed`。
- `target.position` 限制在当前位置附近，避免突变。
- `target.yawAngle` 由 yaw speed 积分得到。

3. LQR balance

- `lqr_k(legLength)` 根据腿长计算增益。
- LQR 输出 wheel torque 和 hip torque component。

4. PID auxiliary loops

- Yaw PID: 转向。
- Roll PID: 左右腿力差。
- Leg length PID: 虚拟腿长控制。
- Leg angle PID: 左右腿角差 / 交叉步。

5. VMC mapping

- `leg_vmc_conv(force, torque, joint angles)` 把虚拟腿力和髋部力矩转成两个关节电机扭矩。

6. Ground/contact and safety states

- 触地检测。
- 离地时关闭轮电机。
- 落地缓冲。
- Balance off 归零。
- 角度保护。

必须配图：

- Control block diagram
- State machine diagram

### 3.6 ROS 2 vision, WiFi TCP and input arbitration

这一节讲通信和输入，不要混到 LQR 里。

Firmware command interface：

- UART2 和 WiFi TCP 都调用同一套 command parser。
- WiFi TCP 通过 `Serial_InjectCommandLine()` 注入命令。

Direct commands：

- `DRIVE,<speed_mm_s>,<yaw_mrad_s>`
- `YAWRATE,<mrad_s>`
- 直接改 target，不进队列。
- 适合连续遥操作。

Queue commands：

- `FORWARD`
- `BACKWARD`
- `LEFT`
- `RIGHT`
- `JUMP`
- `CROSSLEG`
- `INCREASELEGLENGTH`
- `DECREASELEGLENGTH`

Arbitration：

- 队列运行时 BLE 和 direct yaw/drive 被抑制。
- PC tool 启动时发送 `BLE_DISABLE`。
- PC tool 退出时发送 `DRIVE,0,0 -> YAWRATE,0 -> BLE_ENABLE`。

Watchdogs：

- 500 ms 没有新 `DRIVE` 就停车。
- 500 ms 没有新 `YAWRATE` 就清零 yaw。
- 1500 ms TCP idle / client drop / WiFi lost 触发 full stop。

Vision bridge：

- Camera micro-ROS → `/espRos/esp32camera`
- ROS 2 node → MediaPipe
- Command encoder → TCP transport
- `dry_run` 防止未验证时真实发命令
- `stunt_armed` 防止误触发危险动作

## 实验设计优化思路

优秀 report 的实验不是“测很多”，而是每个实验都能清楚回答一个 objective，并且数据可复现。当前最值得改的不是增加很多实验，而是提高实验证据质量。

### 最高优先级：统一日志系统

如果只能改一个程序，就先改 logger。现在最大的证据风险是实验数据不够结构化。

建议新增 `Telemetry_Task` 或扩展现有 `Log_Task`，输出 CSV：

```text
t_ms,pitch_deg,roll_deg,yaw_deg,pitch_rate,roll_rate,yaw_rate,
leg_L_m,leg_R_m,wheel_L_rad_s,wheel_R_rad_s,
cmd_vx,cmd_yaw,src,balance_enabled,queue_state,event
```

建议新增命令：

```text
LOG_ON
LOG_OFF
MARK,<experiment>,<trial>,<event>
PING,<seq>
```

实验中用 `MARK` 标记关键事件：

```text
MARK,E2,trial_03,impact
MARK,E4,trial_01,step_start
MARK,E6,trial_04,tcp_disconnect
```

这样 Results 章节可以画真正的时间序列，而不是只靠文字描述。

### O1 控制实验增强

O1 的核心是证明 LQR/PID/VMC 控制有效。建议保留并强化以下实验。

| 实验 | 改进点 | 为什么加分 |
|---|---|---|
| E1 Static balance | 3 次重复，每次 60 s，给 RMS、peak、drift | 证明基础稳定性 |
| E2 Impulse recovery | 用固定摆锤或固定推法，加 `MARK impact` | 扰动可重复，恢复时间可信 |
| E3 Leg length variation | min/mid/max 腿长下重复 E2 | 证明你理解轮腿重心变化代价 |
| E4 Teleop step response | `DRIVE` 阶跃输入，记录 pitch 偏移和速度响应 | 证明运动命令和平衡耦合 |
| E8 Loop jitter | 记录 `CtrlBasic_Task` 每周期 `micros()` 差值 | 证明实时性 |

最能拉分的是 E2 和 E8：

- E2 给“机器人真的能稳”的证据。
- E8 给“实时控制设计合理”的证据。

### O2 通信和视觉实验增强

O2 不要做“ESP32 micro-ROS RTT”，因为当前底盘不是 micro-ROS node。应该做当前真实系统的实验。

建议新增 `PING,<seq>`：

- PC 发 `PING,123`。
- ESP32 回 `PONG,123,<esp_ms>`。
- PC 记录 RTT。
- 画 RTT CDF。

这比用串口日志推断延迟更干净。

建议实验：

| 实验 | 内容 | 指标 |
|---|---|---|
| E5 TCP latency | `PING/PONG` 或安全命令 ACK | median, p95, p99 RTT |
| E6 Watchdog | 停止发送、关闭 socket、断 WiFi | stop latency, success rate |
| E7 Vision | camera topic hz + gesture event log | FPS, event success rate, command delay |

E6 很重要。它体现工程安全性：即使 WiFi、PC 或视觉桥挂了，机器人也会停。

### O3 输入仲裁实验

这是项目的工程亮点，建议单独做一个小表格实验。

| 场景 | 预期结果 |
|---|---|
| BLE 正在控制，PC 工具启动发送 `BLE_DISABLE` | BLE 不再覆盖 target |
| Queue 正在运行，发送 `DRIVE` | `DRIVE ignored: command queue busy` |
| TCP client 断开 | 自动 `DRIVE,0,0 + YAWRATE,0 + QUEUE_STOP` |
| Vision `dry_run=true` | 只打印，不真实发命令 |
| `stunt_armed=false` | `JUMP/CROSSLEG` 被阻止 |

这个实验不需要复杂数据，但非常适合放在 §4.4，用 pass/fail 表展示安全设计。

### 不建议优先做的实验

除非核心数据都采完，否则不要优先做：

- ROS 2 QoS best effort vs reliable 控制命令对比，因为当前控制命令不是 ROS 2 topic。
- 复杂视觉 latency breakdown，因为没有同步时钟会很难讲清。
- 大量不同地面测试，时间成本高且风险大。
- 关闭安全保护做对照实验，不安全，也不值得。

### 最优实验执行顺序

如果时间有限，按这个顺序做：

1. 改 CSV logger + event marker。
2. E8 loop jitter。
3. E1 static balance。
4. E2 impulse recovery。
5. E5 TCP latency with `PING/PONG`。
6. E6 watchdog/fault injection。
7. E7 vision event log。
8. E3/E4 有时间再做。

这样 Results 章节会覆盖四类关键证据：

- 控制性能
- 实时性
- 通信安全
- 视觉遥操作可行性

不要按章节顺序从 Introduction 开始写。最稳的顺序是先写确定的工程事实，再写实验结果，再回头写背景和总结。

### 1. 先写 Methods

先把“你做了什么”写清楚，因为 Methods 最确定、最不依赖最终实验结果。

建议顺序：

```text
3.2 System Architecture
3.4 Perception and State Estimation
3.5 Control Architecture and Algorithms
3.6 ROS 2 Vision, WiFi TCP and Input Arbitration
3.3 Mechanical and Electrical Design
3.1 Top-level Requirements
3.7 Summary
```

原因：

- `3.2` 是整篇报告的骨架。
- `3.4` 和 `3.5` 是控制核心。
- `3.6` 是你项目的工程扩展和安全亮点。
- `3.3` 硬件设计可以后补，但要和 Methods 前文保持一致。
- `3.1` requirements 最好等架构写清后再反向整理。

### 2. 再写 Results and Discussion

有什么数据就写什么，不要先承诺结论。先做图表，再围绕图表写文字。

建议顺序：

```text
4.1 Test Plan
4.2 System Bring-up and Baseline
4.3 Balance and Motion Control Performance
4.4 WiFi TCP, Camera micro-ROS and Vision Teleop Performance
4.5 Summary
```

写法：

- 先定每张图和每张表。
- 每张图后按“现象 → 原因 → 意义”解释。
- 每个实验都要回扣 O1/O2/O3。
- 不要把没有测到的数据写成已经完成。

### 3. 再写 Literature Review

这时你已经知道 Methods 和 Results 需要哪些背景，就不会写散。

建议顺序：

```text
2.2 Wheel-legged and Self-balancing Robotics
2.3 Control Strategies Comparison
2.4 Embedded ROS, WiFi Teleoperation and Lossy Networks
2.5 Summary
2.1 Introduction
```

原因：

- 先写具体主题，再写综述 introduction 更容易。
- Literature Review 必须服务 Methods 和 Results。
- 不要逐篇摘要，要做主题比较。

### 4. 再写 Introduction

Introduction 要反过来服务整篇报告，所以等 Methods、Results、Literature Review 大致稳定后再写。

建议顺序：

```text
1.1 Background and Motivation
1.2 Aims and Objectives
1.3 Report Structure
1.4 Project Management
```

重点：

- `1.2` 的 objectives 必须和 §4 的 evidence 对上。
- `1.4` Project Management 必须引用附录中的 Gantt、Risk Register、CPD 和 H&S。

### 5. 最后写 Conclusion 和 Abstract

建议顺序：

```text
5.1 Conclusions
5.2 Future Work
Abstract
```

Conclusion：

- 逐条回应 O1/O2/O3。
- 写清楚完成了什么、没完成什么。
- 不要重新复述 Methods。

Future Work：

- 必须从 Results 的局限自然推出。
- 不要写空泛愿望。

Abstract：

- 最后写。
- 必须包含 problem、motivation、method、key results、limitations、contribution。
- 不要放引用。
- 不要出现未定义缩写。

### 6. 最后处理 Front Matter / Appendices / Formatting

最后再处理：

- Title page
- Declaration of Originality
- Intellectual Property Statement
- Contents
- List of Figures
- List of Tables
- Abbreviations / Nomenclature
- References
- Appendices
- README copy / repo link / third-party license table

最终检查：

- 所有 `WRITE HERE` 和占位文字清空。
- 所有图表在正文中先引用再出现。
- 所有公式变量首次出现有定义。
- IEEE 引用格式一致。
- 正文不超过 35 页。
- PDF 能打开，目录页码正确。
- repo 在提交时 public。
- 保存 Canvas submission confirmation。

### 最实际的写作顺序总表

```text
3.2 → 3.4 → 3.5 → 3.6 → 3.3 → 3.1 → 3.7
4.1 → 4.2 → 4.3 → 4.4 → 4.5
2.2 → 2.3 → 2.4 → 2.5 → 2.1
1.1 → 1.2 → 1.3 → 1.4
5.1 → 5.2
Abstract
Appendices + formatting
```

核心原则：

> 先写确定的工程事实，再写证据，再写背景和总结。
